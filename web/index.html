tinyMem

Local, enforced memory for AI agents that work in real repositories.

tinyMem makes agents recall context, verify facts against code, and act only when evidence exists.

Get Started GitHub Repository

Why tinyMem Exists

AI agents fail in real projects for predictable reasons: memory drift, unverifiable assumptions, and actions taken without checking the repository state.

tinyMem was built to fix that. Not by giving models more context, but by enforcing how agents are allowed to reason and act.

With tinyMem, an agent must recall memory, verify claims against code, and produce evidence before executing changes. If it skips a step, the response is invalid by design.

# Install tinyMem (macOS / Linux)
os="$(uname -s | tr '[:upper:]' '[:lower:]')"
arch="$(uname -m)"; [ "$arch" = "x86_64" ] && arch="amd64"; [ "$arch" = "aarch64" ] && arch="arm64"
curl -L "https://github.com/daverage/tinyMem/releases/latest/download/tinymem-${os}-${arch}" -o tinymem
chmod +x tinymem && sudo mv tinymem /usr/local/bin/

# Initialize
tinymem health

# Run in MCP mode for IDEs
tinymem mcp
> MCP server listening on stdio

tinyMem runs locally, stays project-scoped, and can operate entirely without cloud dependencies or complex embeddings. Optional semantic search is available for users who want vector-based recall.

Evidence-Backed Memory

Memory entries are typed, scoped, and verifiable. Claims can be validated against files, grep results, test runs, or command exits. Guessing is replaced with proof.

The Ralph Loop

An optional autonomous repair loop that retries until evidence predicates pass. Tests fail, memory is recalled, fixes are applied, and execution continues only when verification succeeds.

tinyTasks

A file-authoritative task ledger. Agents must update tinyTasks.md so progress is explicit, reviewable, and persistent across sessions.

The Control Protocol

tinyMem enforces a fixed operational sequence. Agents cannot skip steps or invent state.

1
Recall
Relevant project memory is loaded before reasoning begins.
2
Verify
Existing claims are checked against repository evidence.
3
Lock Tasks
Multi-step work is bound to a durable task ledger.
4
Act & Record
Execution occurs, followed by validated memory write-back.

MCP Server

tinyMem exposes memory tools via the Model Context Protocol. Works with MCP-enabled clients such as Claude Desktop, Cursor, and VS Code adapters.

HTTP Proxy

OpenAI-compatible HTTP proxy for SDKs. Injects verified context into requests when configured, without modifying application code.

Agent Contracts

Opinionated starter policies for Claude, Gemini, Qwen, and others. Define how agents must use memory, tasks, and evidence in your repo.

Built for Reliability

tinyMem is designed for production workflows where confidence matters more than creativity.

  • Chain-of-Verification Memory writes and recalls can be filtered through verification passes to reduce hallucinations.
  • Recall Prioritization Facts and constraints are surfaced before decisions or notes.
  • Evidence Predicates Truth checks via file_exists, grep_hit, cmd_exit0, and test_pass.
  • Token Efficiency Surgical context injection replaces brute-force prompt stuffing.
